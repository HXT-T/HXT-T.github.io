# HXT 个人博客完善设计

日期：2026-07-12  
状态：已由用户确认

## 1. 目标与边界

将现有 Jekyll + Chirpy 博客完善为属于 HXT 的长期互联网空间。网站以兴趣、真实表达、个人记录和持续积累为中心，不以招聘展示、商业合作或流量增长为目标。先在 1～2 周内完成稳定、舒适的标准版，再根据真实需要逐步增加独立域名、数据分析或自动化。

页面布局、配色与视觉风格由站长自主设计。AI 负责解释、生成备选实现、排错与审查，不代替审美判断和个人表达。第一阶段不引入 Vue 或大型 UI 组件库。

## 2. 信息架构

一级入口收敛为：

- 首页：自我介绍、最近书写、兴趣入口、近期动态和随机探索。
- 文章：技术、学习、生活与思考，通过分类和标签辅助浏览。
- 项目或创造：记录自己愿意公开的作品、实验、折腾过程与未完成想法，不要求包装成成果展示。
- 动态：近期学习、计划与简短更新。
- 关于：个人介绍、价值观、兴趣、当前关注，以及愿意公开的联系渠道。

标签、时间线和分类作为文章辅助入口；“相遇”和“赞助”降为二级入口或页脚内容。

首页访问顺序：这里是谁的空间 → 最近在写什么 → 长期喜欢什么 → 最近在做什么 → 随意继续探索。

## 3. 内容体系

四条长期内容主线：

1. 技术实践：项目过程、故障排查、架构与工具经验。
2. 学习记录：课程、阅读、实验和阶段性学习路线。
3. 生活与思考：阅读、旅行、摄影、成长和长期观点。
4. 创造与实验：记录项目、工具、网页、摄影、手工作品、失败尝试和仍在生长的想法，重点是过程与个人意义，而不是证明能力。

标准版准备六篇能体现兴趣范围的内容：两篇技术或创造记录、一篇深入排错或实验、一篇学习总结、一篇生活或成长复盘，以及一篇“关于我与为什么建立这个空间”。它们不必覆盖所有能力，只需让这个空间开始具有真实内容和个人气质。

AI 可以整理提纲、润色、检查 Markdown 和生成标题摘要；站长必须提供真实事实、技术判断、最终观点，并逐段审核 AI 内容。

## 4. 视觉与自主学习策略

基准方向为“克制的技术感与温和的个人表达”，但最终视觉规则由站长决定。设计系统至少定义：主色、强调色、背景色、文本色、字体、内容宽度、圆角、间距、阴影和图片风格。

学习顺序：HTML → CSS 盒模型 → Flexbox/Grid → 响应式 → SCSS → 开发者工具 → Jekyll/Liquid → 原生 JavaScript → 无障碍 → 设计系统。

普通样式集中在 `_sass/addon/custom.scss`。先调整正文、头像、侧边栏、链接、标题、图片和代码块，再实现首页与项目页。尽量不覆盖 Chirpy 核心布局和 JavaScript，以降低升级成本。

每次设计至少测试 320、375、768、1024 和 1440px，并检查浅色、深色、键盘访问、文本对比度、图片替代文本和减少动态效果设置。

## 5. 技术架构

保留当前架构：

- Jekyll 4.3
- Chirpy 6.5
- Ruby 3.2（CI）
- GitHub Pages
- `main` 保存源码
- GitHub Actions 构建 `_site`
- `gh-pages` 保存发布产物

不手动编辑 `_site` 或 `gh-pages`。主题升级前先在分支中构建和检查。复杂交互工具未来可作为独立 Vue 应用部署并从博客链接，不迁移整个博客。

## 6. Obsidian 发布流程

本地仓库 `D:\Az\HXT-T.github.io` 同时作为 Obsidian Vault：

```text
.obsidian/          Obsidian 通用设置
_drafts/            草稿（公开仓库中不得存私人内容）
_posts/             已发布文章
_templates/         文章模板
assets/img/posts/   文章图片
```

建议插件：Obsidian Git、Templater、Linter。关闭 Wiki 链接，使用标准 Markdown 链接。图片使用 `/assets/img/posts/...` 的站点根路径。

发布流程：Git Pull → `_drafts` 写作 → AI 辅助检查 → 人工核实事实和隐私 → 改名并移入 `_posts` → Jekyll 本地预览 → 查看差异 → Commit → Push → 检查 Actions 和线上页面。

Obsidian Git 初期采用手动拉取、提交与推送，避免自动公开未完成草稿。忽略 `.obsidian/workspace.json`、缓存、`_site`、Jekyll 缓存和本地 Ruby 依赖。

## 7. SEO、评论与数据

保留 `jekyll-sitemap`、`jekyll-feed` 和 `jekyll-seo-tag`。每篇文章提供准确标题、描述、分类、标签和图片替代文本，并建立相关旧文与项目的内部链接。

标准版接入 Google Search Console，提交 `/sitemap.xml`。第一阶段不额外安装访问统计；确有决策需求后再选择分析工具并补充隐私说明。

继续使用现有 Giscus 配置，发布前确认 Discussions、Giscus App、分类和文章底部评论正常。文章发布后不随意更改永久链接，避免评论映射丢失。

独立域名放在第二阶段：验证域名 → 添加到 GitHub Pages → 配置 DNS → 开启 HTTPS → 更新 `url` → 检查 canonical、sitemap、评论和图片 → 更新 Search Console。

## 8. 安全与隐私

仓库和网站不得包含密码、Token、密钥、身份证件、精确住址、实时行程、内部资料、他人隐私或未脱敏日志。发布截图前检查浏览器标签、终端历史、路径、邮箱、IP 和通知。

推送前必须检查 Git 差异。AI 可帮助识别潜在敏感信息，但是否公开、域名购买、DNS、账号权限、分析范围和评论治理必须由站长决定。

## 9. 验证与维护

每次发布至少执行生产构建，并检查：

- Jekyll 构建成功。
- Front Matter 可解析。
- 图片、链接、目录和代码高亮正常。
- 手机与桌面布局正常。
- 浅色和深色模式可读。
- 无敏感信息或 AI 虚构内容。
- GitHub Actions 成功，线上页面可访问。

每月检查一次断链、依赖更新、Actions、评论、Search Console 索引和代表内容；先理解变更再升级主题或依赖。

## 10. 分阶段交付

第一周：建立 Obsidian Vault、模板与 Git 工作流；掌握 HTML/CSS 和开发者工具。  
第二周：完善身份、关于页、分类和两篇代表文章；调整正文与侧边栏。  
第三至四周：设计并实现首页、兴趣入口、创造记录、响应式和深色模式；按个人意愿决定是否接入 Search Console。
第二个月起：形成适合自己的写作节奏，继续积累文章和创造记录，并根据真实需要决定独立域名、统计或复杂交互。
